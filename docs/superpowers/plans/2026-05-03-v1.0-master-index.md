## Estimated wall-clock effort (with parallel subagents)

These assume the executor uses `superpowers:subagent-driven-development` with the parallel groups above. Sequential time would be roughly 3–5× longer.

| Phase | Parallel | Sequential equivalent |
|---|---|---|
| 0 | 1–2 hours | 4–6 hours |
| 1 | 2–4 hours | 8–12 hours |
| 2 | 4–8 hours | 24–32 hours |
| 3 | 3–5 hours | 12–18 hours |
| 4 | 8–16 hours (skill evals dominate) | 50–80 hours |
| 5 | 3–6 hours | 15–25 hours |
| 6 | 2–4 hours (gated by CI) | 6–10 hours |
| 7 | 1–2 hours | 2–3 hours |
| **Total** | **24–47 hours** | **121–186 hours** |

The bulk of Phase 4's time is irreducible — generating per-skill eval baselines requires LLM calls that take seconds each × ~150 skills × 5–10 cases each = thousands of API calls. Parallelism helps with the *authoring*; the API calls themselves are bottlenecked by Anthropic's rate limits.

## Critical-path tasks (don't parallelize away)

A few tasks are intentionally sequential because they're invariant-gates:

- Phase 1 Task 7 (write canonical `marketplace.json`) — every Layer A test depends on it
- Phase 2 Tasks 1–3 (shared helpers) — every translator depends on them
- Phase 3 Tasks 1–3 (manifest, detectors, github) — every CLI command depends on them
- Phase 4 Task 3 (first eval pattern) — every per-skill eval is a copy of this template
- Phase 6 Task 1 (cut release branch) — triggers the integration matrix

If you parallelize past these, you'll create churn (subagents fixing the same upstream issue independently).
