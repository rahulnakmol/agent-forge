# GitOps with Flux v2

GitOps for AKS and Container Apps is non-negotiable. The cluster pulls desired state from Git; the pipeline commits manifests and never runs `kubectl apply` directly. Flux v2 is the default for AKS; the `microsoft.flux` cluster extension is managed by Azure and integrates with Azure Policy compliance reporting.

---

## Core Concepts

| Concept | Description |
|---|---|
| `GitRepository` | Source controller CRD; watches a Git repo/branch/tag and produces a versioned artifact |
| `Kustomization` | Kustomize controller CRD; applies a path from a `GitRepository` artifact to the cluster |
| `HelmRelease` | Helm controller CRD; deploys a Helm chart from a `HelmRepository` or `GitRepository` source |
| `fluxconfig-agent` | Polls Azure for `fluxConfigurations` resources; syncs desired state into the cluster |
| `fluxconfig-controller` | Watches `FluxConfig` CRs; creates `GitRepository` + `Kustomization` objects |
| `microsoft.flux` extension | The Azure-managed Flux cluster extension; installed via `az k8s-extension create` or automatically on first `fluxConfiguration` |

The N-2 version policy applies to `microsoft.flux`; run the latest release to receive security patches. The extension installs the Source, Kustomize, Helm, and Notification controllers by default. Optionally enable the `image-automation` and `image-reflector` controllers for automated image tag updates.

## Flux Extension Installation

```bash
# Install the microsoft.flux extension on an existing AKS cluster
az k8s-extension create \
  --resource-group rg-aks-prod \
  --cluster-name aks-prod \
  --cluster-type managedClusters \
  --name flux \
  --extension-type microsoft.flux \
  --scope cluster

# Verify installation
az k8s-extension show \
  --resource-group rg-aks-prod \
  --cluster-name aks-prod \
  --cluster-type managedClusters \
  --name flux
```

## GitRepository CRD

```yaml
# infrastructure/base/flux-system/gitrepository.yaml
apiVersion: source.toolkit.fluxcd.io/v1
kind: GitRepository
metadata:
  name: gitops-config
  namespace: flux-system
spec:
  interval: 1m0s           # Poll interval -- 1 minute for responsive reconciliation
  url: https://github.com/my-org/gitops-config
  ref:
    branch: main
  secretRef:
    name: gitops-config-auth   # Kubernetes Secret with deploy key or token
  timeout: 60s
  ignore: |
    # Ignore CI pipeline files -- they don't affect cluster state
    .github/
    docs/
    *.md
```

For private repositories, create a deploy key and reference it as a Kubernetes Secret:

```bash
# Generate deploy key
ssh-keygen -t ed25519 -C "flux@aks-prod" -f ./deploy-key -N ""

# Add public key to the GitHub repo as a deploy key (read-only is sufficient)

# Create the Secret in the cluster
kubectl create secret generic gitops-config-auth \
  --namespace flux-system \
  --from-file=identity=./deploy-key \
  --from-file=identity.pub=./deploy-key.pub \
  --from-literal=known_hosts="$(ssh-keyscan github.com)"
```

## Kustomization CRDs: Sequenced Deployment

The `infrastructure` Kustomization must reconcile before `apps`. The `dependsOn` field enforces this.

```yaml
# clusters/aks-prod/infrastructure.yaml
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: infrastructure
  namespace: flux-system
spec:
  interval: 5m0s
  sourceRef:
    kind: GitRepository
    name: gitops-config
  path: ./infrastructure/overlays/production
  prune: true            # Remove resources deleted from Git
  wait: true             # Wait for all resources to reach Ready state
  timeout: 5m0s
  healthChecks:
    - apiVersion: apps/v1
      kind: Deployment
      name: ingress-nginx-controller
      namespace: ingress-nginx
```

```yaml
# clusters/aks-prod/apps.yaml
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: apps
  namespace: flux-system
spec:
  interval: 1m0s
  sourceRef:
    kind: GitRepository
    name: gitops-config
  path: ./apps/overlays/production
  prune: true
  wait: true
  timeout: 5m0s
  dependsOn:
    - name: infrastructure    # Apps deploy only after infrastructure is healthy
  postBuild:
    substituteFrom:
      - kind: ConfigMap
        name: cluster-vars    # Inject cluster-specific values (region, registry URL)
```

## Creating a fluxConfiguration via Azure CLI

This creates the Azure-layer resource that drives the in-cluster Flux objects:

```bash
az k8s-configuration flux create \
  --resource-group rg-aks-prod \
  --cluster-name aks-prod \
  --cluster-type managedClusters \
  --name cluster-config \
  --namespace flux-system \
  --scope cluster \
  --url https://github.com/my-org/gitops-config \
  --branch main \
  --kustomization \
    name=infrastructure \
    path=./infrastructure/overlays/production \
    prune=true \
    timeout=5m \
  --kustomization \
    name=apps \
    path=./apps/overlays/production \
    prune=true \
    timeout=5m \
    depends_on=infrastructure
```

## Image Automation Controllers

Enable the `image-reflector` and `image-automation` controllers to automatically update image tags in the config repo when a new image is pushed to ACR.

```bash
# Enable image automation at extension install time
az k8s-extension create \
  --resource-group rg-aks-prod \
  --cluster-name aks-prod \
  --cluster-type managedClusters \
  --name flux \
  --extension-type microsoft.flux \
  --scope cluster \
  --configuration-settings \
    image-automation-controller.enabled=true \
    image-reflector-controller.enabled=true
```

```yaml
# Image policy -- track the latest semver tag on the main channel
apiVersion: image.toolkit.fluxcd.io/v1beta2
kind: ImageRepository
metadata:
  name: my-app
  namespace: flux-system
spec:
  image: myacr.azurecr.io/my-app
  interval: 5m0s
  secretRef:
    name: acr-credentials
---
apiVersion: image.toolkit.fluxcd.io/v1beta2
kind: ImagePolicy
metadata:
  name: my-app
  namespace: flux-system
spec:
  imageRepositoryRef:
    name: my-app
  policy:
    semver:
      range: '>=1.0.0'
---
apiVersion: image.toolkit.fluxcd.io/v1beta1
kind: ImageUpdateAutomation
metadata:
  name: my-app
  namespace: flux-system
spec:
  interval: 5m0s
  sourceRef:
    kind: GitRepository
    name: gitops-config
  git:
    checkout:
      ref:
        branch: main
    commit:
      author:
        email: fluxbot@my-org.com
        name: Flux Bot
      messageTemplate: |
        chore(gitops): update my-app image to {{ .NewImage }}
    push:
      branch: main
  update:
    path: ./apps/overlays/production/my-app
    strategy: Setters
```

In the deployment manifest, mark the image tag for automated update:

```yaml
# apps/overlays/production/my-app/patch-image.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  template:
    spec:
      containers:
        - name: my-app
          image: myacr.azurecr.io/my-app:1.3.2   # {"$imagepolicy": "flux-system:my-app"}
```

## Monitoring Flux Status

```bash
# Check overall reconciliation status
flux get all --namespace flux-system

# Check a specific Kustomization
flux get kustomization apps --namespace flux-system

# Watch for reconciliation events
flux events --namespace flux-system --watch

# Force immediate reconciliation (useful after a manual Git push)
flux reconcile source git gitops-config --namespace flux-system
flux reconcile kustomization apps --namespace flux-system

# Suspend reconciliation (e.g., during maintenance)
flux suspend kustomization apps --namespace flux-system

# Resume
flux resume kustomization apps --namespace flux-system
```

## Multi-Tenancy Pattern

Multiple application teams share one AKS cluster. Each team has a dedicated namespace and its own `fluxConfiguration` scoped to that namespace:

```bash
# Team A -- scoped to team-a namespace
az k8s-configuration flux create \
  --resource-group rg-aks-prod \
  --cluster-name aks-prod \
  --cluster-type managedClusters \
  --name team-a-config \
  --namespace team-a \
  --scope namespace \
  --url https://github.com/my-org/team-a-config \
  --branch main \
  --kustomization name=team-a-apps path=./apps prune=true
```

Each namespace-scoped configuration can only manage resources in its own namespace; the Flux operator enforces this boundary via Kubernetes RBAC.

## Pipeline Integration: CI Commits, GitOps Deploys

The CI/CD pipeline's deploy job commits the new image tag to the config repo. Flux reconciles the change within the configured interval.

```yaml
# GitHub Actions: update image tag in config repo after push to ACR
- name: Update GitOps config repo
  env:
    GIT_AUTHOR_EMAIL: ci-bot@my-org.com
    GIT_AUTHOR_NAME: CI Bot
  run: |
    git clone https://x-access-token:${{ secrets.GITOPS_PAT }}@github.com/my-org/gitops-config /tmp/gitops
    cd /tmp/gitops

    # Update image tag using kustomize edit
    cd apps/overlays/production/my-app
    kustomize edit set image myacr.azurecr.io/my-app=myacr.azurecr.io/my-app:${{ github.sha }}

    git add .
    git commit -m "chore(gitops): deploy my-app ${{ github.sha }} to production"
    git push
```

The GitOps PAT for the config repo commit is a dedicated fine-grained PAT scoped to the config repo with `contents: write` only. Store it in the GitHub Environment secrets for the `production` environment.
