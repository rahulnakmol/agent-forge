> **Architect Notes**: The Master Data Management, or MDM, Design and Governance asset is the third of three assets in the Performance Insights & Data layer of the Target Operating Model. It provides a structural guide of how master data is designed and governed, enabling you to manage and govern master data during and after a Powered implementation. The MDM asset augments and aggregates other components of the Target Operating Model including the Process, People, Technology, Performance Insights & Data, and Governance design layers to explain what drives the way master data looks and how it is structured. As you can see here, while the Master Data Management template resides under Performance Insights & Data, it impacts several other areas, including Process, Technology, and Governance. The Tech Agnostic MDM asset addresses the business requirements and functional solutions for key master data domains. The Platform Specific MDM assets capture the technical solutions required to address the business requirements for key master data domains.

## MDM Definition & Design Principles

| Master Data Domain Name: | Supplier Master Data |
| --- | --- |
| Primary Use of this Master Data: | Aggregate Supplier data from multiple sources, enrich data and maintain an accurate 360 Supplier view ensuring data integrity. The results lead to enhanced user/Supplier experience and gaining competitive advantage in your industry |

| Design Principles Guiding Structure of this Master Data (Both General and Platform Specific): | 1. Common/standard data model and data structure requirements so that master data is stored consistently in all applications across the organization, creating a centralized single version of truth around your master data
2. Suppliers should be responsible for maintaining their own master data where applicable 3. Supplier master data should be controlled by a centralized team and all changes to the supplier master data should be subject to Procurement/business review (as applicable)
4. Quality of the Supplier master file is regularly reviewed and cleansed to remove inactive, duplicate and out of date records 5. Use third-party data validation and enrichment for addresses, interests & risk profile
6. Supplier records conform to a consistent naming convention that is applied across the organization
7. Cleansed and consolidated master data changes are shared/replicated automatically across all subscribing applications 8. Self service dashboards should be used to slice and dice supplier related information to facilitate flexible, ad-hoc reporting and minimize overall reports |  |  |  |  |  |  |

## Master Data Reports & Master Data Controls

| Master Data - Reports |  |
| --- | --- |
| Reporting Requirements | MDM Functional Solution |
| Ability to report on all information held in the supplier master data record to support master data needs | Supplier Master Listing (summary or detail) |
| Ability to report spend by purchasing categories and sub-categories | Spend by Category |
| Ability to report on consolidated supplier spend and supplier performance to support contract management | Supplier Relationship Listing Report (include hierarchy) |
| Ability to identify duplicate supplier record to minimize maintenance requirements | Duplicate Supplier Report |
| Ability to report on risk profile of supplier base to inform decisions | Supplier Risk Report |
| Ability to identify opportunities to reduce costs through discounting taken/offered by suppliers | Suppliers Using/Offering Dynamic Discounting |
| Ability to report on spend by interest groups (e.g. ethnic/minority suppliers) to meet any potential regulatory requirements | Supplier Ethnic/Minority Ownership & Spend Report |
| Ability to report on contractual & process compliance of the supplier base to identify opportunities to deliver benefit | Supplier Compliance Report (Contracts, PO/Non-PO, Invoicing, Payment Terms, etc.) |

| Master Data - Controls |  |
| --- | --- |
| Control Requirement | MDM Functional Solution |
| 1. Accuracy - The data values match with source of correct/true information | Defined Sources of data (with data validation from an authorized user) with clear audit trail and reconciliation of data on a monthly basis |
| 2. Consistency - Accordance/conformity of metadata (i.e., data definition/meaning/understanding | Cleary defined definition and nomenclature of data, with periodic controls and checks |
| 3. Compliance - Audit trail for all supplier Setup and changes | Clear and retrievable audit trail must be maintained |
| 4. Supplier ownership - Supplier is the ultimate owner of their Master Data | Supplier must initiate any setup/change and provide relevant documentation |
| 5. Embedded policies - Policies are "hardcoded" into the system during supplier setup | System must not allow users to complete critical fields without predetermined entries |
| 6. Data enhancement - Validation/Approval needs to take place in any data enhancement | Workflow approval must be used for any supplier master data enhancement |
| 7. Supplier Risk profile - We will need to monitor the suppliers risk profile and our exposure | In system or Third party feed needs to be use to provide comprehensive supplier risk profiles |
| 8. Segregation of Duty - We need to ensure that any person changing data is not the same that approves the change | Cleary defined roles with respective accesses must be maintained |
