# Azure DevOps CSV Import Format

## Overview

Azure DevOps supports bulk import of work items via CSV files. This document
specifies the exact format required for importing backlog items generated by the
PM Backlog Generator skill.

---

## CSV Column Headers

The CSV file MUST use the following column headers in this exact order:

```csv
Work Item Type,Title,State,Assigned To,Area,Iteration,Description,Acceptance Criteria,Priority,Risk,Story Points,Effort,Business Value,Time Criticality,Value Area,Parent,Tags
```

### Column Definitions

| Column           | Required | Applies To                  | Format                              |
|------------------|----------|-----------------------------|-------------------------------------|
| Work Item Type   | Yes      | All                         | Exact WIT name (see below)          |
| Title            | Yes      | All                         | Plain text, max 255 chars           |
| State            | Yes      | All                         | "New" for all generated items       |
| Assigned To      | No       | All                         | Email or display name (leave blank) |
| Area             | Yes      | All                         | Project\Team path                   |
| Iteration        | Yes      | All                         | Project\Release path                |
| Description      | Yes      | All                         | HTML content (see encoding rules)   |
| Acceptance Criteria | Cond  | User Story, Tech Story      | HTML content (see encoding rules)   |
| Priority         | Yes      | All                         | 1, 2, or 3                          |
| Risk             | Cond     | Epic, Feature, Story, Risk  | Integer or letter (see WIT ref)     |
| Story Points     | Cond     | User Story, Tech Story      | Fibonacci number                    |
| Effort           | Cond     | Feature                     | Numeric (hours)                     |
| Business Value   | Cond     | Feature                     | 1-100                               |
| Time Criticality | Cond     | Feature                     | 1-100                               |
| Value Area       | Cond     | Epic, Feature, Story        | "Business" or "Architectural"       |
| Parent           | Cond     | Feature, Story, Risk        | Title of parent work item           |
| Tags             | No       | All                         | Semicolon-separated                 |

### Allowed Work Item Type Values

- `Epic`
- `Feature`
- `User Story`
- `Technical User Story`
- `Risk`
- `Impediment`
- `Continuous Improvement Item`

---

## Parent Column Rules

The Parent column is critical for establishing hierarchy:

1. The value MUST be the exact **Title** of the parent work item (not an ID).
2. The parent work item MUST appear in the CSV **before** the child work item.
3. If a work item has no parent (e.g., Epic, Impediment), leave the Parent column
   empty.
4. ADO matches parent by title within the same CSV import batch. If the title is
   ambiguous (multiple items with the same title), the import may fail.
5. To avoid ambiguity, ensure all work item titles are unique within the CSV file.

### Hierarchy Example

```csv
Work Item Type,Title,State,...,Parent,Tags
Epic,Customer Onboarding Platform,New,...,,
Feature,Self-Service Registration,New,...,Customer Onboarding Platform,
User Story,"As a customer, I want to register with my email",New,...,Self-Service Registration,
Technical User Story,"As a developer, I want to set up OAuth2 provider",New,...,Self-Service Registration,
Risk,Third-party identity provider downtime,New,...,Customer Onboarding Platform,
```

## Multi-Line Content Handling

### HTML in Description and Acceptance Criteria

ADO CSV import supports HTML content in the Description and Acceptance Criteria
fields. Follow these rules:

1. **Wrap the entire field value in double quotes** if it contains commas, newlines,
   or double quotes.
2. **Escape double quotes** within the field by doubling them: `""`.
3. **Use HTML tags** for formatting (not markdown):
   - Paragraphs: `<p>text</p>`
   - Bold: `<strong>text</strong>`
   - Lists: `<ul><li>item</li></ul>` or `<ol><li>item</li></ol>`
   - Headings: `<h3>heading</h3>`
   - Line breaks: `<br/>`
4. **Do not use actual newlines** inside the CSV field value. All HTML must be on
   a single line within the quoted field.

### Example: Description Field

```csv
"<h3>Overview</h3><p>This feature enables customers to self-register on the platform using their email address.</p><h3>Business Context</h3><p>Currently, registration requires manual admin action, which creates a bottleneck.</p>"
```

### Example: Acceptance Criteria Field

```csv
"<ul><li><strong>Given</strong> a new user visits the registration page, <strong>When</strong> they enter a valid email and password, <strong>Then</strong> an account is created and a verification email is sent.</li><li><strong>Given</strong> a user enters an already-registered email, <strong>When</strong> they submit the form, <strong>Then</strong> an error message is displayed.</li></ul>"
```

## Import Order Requirements

Items MUST appear in the CSV in dependency order. Parents must appear before
their children:

1. **Epics** (no parents -- listed first)
2. **Features** (parent = Epic)
3. **User Stories** (parent = Feature)
4. **Technical User Stories** (parent = Feature)
5. **Risks** (parent = Epic or Feature -- after their parent)
6. **Impediments** (standalone -- can appear anywhere, but conventionally at the end)
7. **Continuous Improvement Items** (standalone -- at the end)

### Ordering Algorithm

```
1. Sort all Epics alphabetically by title
2. For each Epic:
   a. List the Epic row
   b. Sort its child Features alphabetically
   c. For each Feature:
      i.   List the Feature row
      ii.  List all child User Stories (sorted by priority, then title)
      iii. List all child Technical User Stories (sorted by priority, then title)
      iv.  List any child Risks
3. List any Epic-level Risks
4. List all Impediments
5. List all Continuous Improvement Items
```

## Validation Checklist Before Import

Before importing the CSV into Azure DevOps, verify:

- [ ] All Work Item Type values match the SAFe process template exactly
- [ ] All parent items appear before their children in the file
- [ ] Parent column values match parent titles exactly (case-sensitive)
- [ ] No duplicate titles exist (to avoid ambiguous parent references)
- [ ] HTML content in Description/AC fields is properly escaped
- [ ] Priority values are 1, 2, or 3 (not text)
- [ ] Story Points use Fibonacci values only
- [ ] Area and Iteration paths exist in the target ADO project
- [ ] State is "New" for all items
- [ ] CSV file is UTF-8 encoded (with BOM for Excel compatibility)
