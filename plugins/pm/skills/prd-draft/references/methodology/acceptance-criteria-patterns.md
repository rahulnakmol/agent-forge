# Acceptance Criteria Patterns

## Purpose

Acceptance criteria (AC) define the conditions that must be true for a user story to be considered complete. They are the contract between product and engineering. If AC are ambiguous, testing is guesswork and disputes are inevitable.

Every acceptance criterion in a PRD generated by this skill uses the Given-When-Then (Gherkin) format.

---

## Given-When-Then Format

```gherkin
Given {precondition -- the starting state of the system}
When {action -- what the user or system does}
Then {expected result -- the observable outcome}
```

### Rules

1. **Given** establishes context: what must be true before the action occurs
2. **When** describes exactly one action (not a sequence of actions)
3. **Then** describes an observable, verifiable outcome
4. Each story should have 3-8 acceptance criteria
5. At least one criterion must cover a failure/edge case

### Simple Example

```gherkin
Given I am a registered user on the login page
When I enter valid credentials and click "Sign In"
Then I am redirected to my dashboard and see my name in the header
```

### Example with Multiple Conditions

```gherkin
Given I am a warehouse manager viewing the inventory dashboard
  And the stock level for item SKU-1234 is below the reorder threshold of 50 units
When the system runs the daily stock check at 06:00 UTC
Then I receive an email alert listing SKU-1234 with current stock level and recommended reorder quantity
  And the alert is logged in the notification center with status "Unread"
```

## Scenario Categories

Every story should include acceptance criteria from multiple categories to ensure comprehensive coverage.

### 1. Happy Path (Required -- at least one per story)

The primary success scenario where everything works as expected.

```gherkin
Given I am an approved vendor in the procurement system
When I submit an invoice with valid PO reference, correct line items, and total under $10,000
Then the invoice is auto-matched to the PO
  And the invoice status changes to "Approved"
  And I receive a confirmation email with estimated payment date
```

### 2. Validation / Boundary Conditions (Required -- at least one per story)

What happens at the edges of valid input.

```gherkin
Given I am creating a new project
When I enter a project name longer than 100 characters
Then the system displays an inline error: "Project name must be 100 characters or fewer"
  And the form is not submitted
  And my other field entries are preserved
```

```gherkin
Given I am uploading a document
When I select a file larger than 25 MB
Then the upload is rejected with message "Maximum file size is 25 MB"
  And no partial upload is stored
```

### 3. Negative / Error Scenarios (Required -- at least one per story)

What happens when things go wrong.

```gherkin
Given I am logged in and submitting a payment
When the payment gateway returns a timeout error
Then the system displays "Payment could not be processed. Please try again."
  And no charge is applied to my account
  And the failed attempt is logged for support investigation
```

```gherkin
Given I am an employee submitting a time-off request
When I request dates that overlap with an already-approved leave period
Then the system displays "These dates overlap with your approved leave from [dates]"
  And the request is not submitted
```

### 4. Authorization / Access Control

Who can and cannot perform the action.

```gherkin
Given I am a team member (not a team lead or admin)
When I attempt to approve a purchase order
Then the "Approve" button is not visible
  And if I access the approval endpoint directly, I receive a 403 Forbidden response
```

### 5. Performance / Non-Functional

Measurable performance expectations.

```gherkin
Given the system has 10,000 active users
When I load the analytics dashboard
Then the dashboard renders completely within 3 seconds
  And all charts display data no older than 5 minutes
```

### 6. State Transition

How the system moves between states.

```gherkin
Given an order is in "Processing" status
When the warehouse confirms all items are packed
Then the order status changes to "Ready for Shipment"
  And the customer receives a notification: "Your order is ready for shipment"
  And the order cannot be cancelled by the customer
```

### 7. Data Integrity

What happens to data during and after the action.

```gherkin
Given I am editing my profile information
When I update my email address and click "Save"
Then a verification email is sent to the new address
  And my profile continues to show the old email until the new one is verified
  And the audit log records the change attempt with timestamp and old/new values
```

## Writing Testable Criteria

### The Testability Checklist

For every acceptance criterion, verify:

- [ ] **Specific**: No vague terms ("fast", "easy", "user-friendly", "appropriate")
- [ ] **Measurable**: Numbers, thresholds, exact text, or observable states
- [ ] **Binary**: Either passes or fails -- no partial credit
- [ ] **Independent**: Does not depend on other criteria being checked first
- [ ] **Automatable**: Could be verified by an automated test (even if manually tested)

### Quantifying Vague Requirements

| Vague | Testable |
|-------|----------|
| "Should be fast" | "Loads in under 2 seconds on 4G connection" |
| "User-friendly interface" | "New users complete the task in under 3 clicks without documentation" |
| "Handles large datasets" | "Processes 1 million records within 60 seconds" |
| "Secure" | "All PII fields are encrypted at rest using AES-256" |
| "Reliable" | "99.9% uptime measured monthly; auto-recovers from single-node failure within 30 seconds" |
| "Mobile responsive" | "All features usable on screens 375px wide and above" |

## Common AC Anti-Patterns

### 1. Implementation-Prescriptive

**Bad:**
```
Given the React component mounts
When the useEffect hook fires
Then the API call returns JSON and setState updates the component
```

**Fix:** Describe the user-observable behavior, not the implementation.

```
Given I navigate to the dashboard
When the page finishes loading
Then I see my recent activity list with the 10 most recent items
```

### 2. Untestable Subjective Criteria

**Bad:**
```
Then the page should look professional and modern
```

**Fix:** Reference a design spec or measurable attributes.

```
Then the page renders according to the approved Figma design (link)
  And all text meets WCAG AA contrast ratio (4.5:1 minimum)
```

### 3. Missing Error Handling

**Bad:** Only happy-path criteria, no failure scenarios.

**Fix:** Always include at least one negative scenario per story. Ask: "What happens when the network fails? When input is invalid? When permissions are denied?"

### 4. Compound Actions in "When"

**Bad:**
```
When I fill in the form, select a file, agree to terms, and click Submit
```

**Fix:** Each "When" should be a single action. Break into multiple criteria or describe the compound action as a scenario.

```
Given I have filled in all required fields
  And I have attached a valid document
  And I have accepted the terms of service
When I click "Submit"
Then the application is created with status "Pending Review"
```

### 5. Missing Preconditions

**Bad:**
```
When I click "Delete"
Then the item is removed
```

**Fix:** What state is the system in? Who am I? What item?

```
Given I am an admin viewing the product catalog
  And the product "Widget X" has no active orders
When I click "Delete" on "Widget X" and confirm the deletion dialog
Then "Widget X" is removed from the catalog
  And a success message "Product deleted" is displayed
```

## AC Count Guidelines

| Story Complexity | Recommended AC Count | Minimum Categories |
|-----------------|---------------------|-------------------|
| S (1-2 days) | 3-5 | Happy path + 1 negative |
| M (3-5 days) | 5-8 | Happy path + negative + boundary |
| L (1-2 weeks) | 8-12 | Happy path + negative + boundary + auth + performance |

If a story needs more than 12 AC, it is likely too large and should be split.
